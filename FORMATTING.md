# Рич-форматирование ответов LOCUS

Как устроены посты бота в Telegram: структура ответа, применяемая разметка и путь
от raw-конспекта до сообщения. Код: `bridge/telegram_out.py`, `bridge/archive.py`,
`bridge/main.py` (`build_digest_from_file`, доставка в `execute_task`).

## 1. Каналы доставки

Один raw-конспект (`vault/raw/YYYY-MM-DD-slug.md`) рождает два поста:

1. **Чат владельца** — выжимка текстом (`build_digest_from_file` → `send_markdown_text`).
2. **Архивный канал** (если задан `ARCHIVE_CHANNEL_ID`) — та же выжимка + строка
   хэштегов из front-matter (`#type` + `#tags`), собирается в `archive.build_channel_post`.

Файлы `.md` в Telegram **не отправляются**: Bot API отдаёт их с кривым MIME
(расширения нет в таблице TDLib), предпросмотр ломается. Отправка файла осталась
только как фолбэк, если выжимку не удалось собрать.

## 2. Транспорт: rich-сообщения Bot API 10.1

- Основной путь — `sendRichMessage` с полем `rich_message: {"markdown": "<текст>"}`:
  Telegram сам парсит Markdown (заголовки, списки, таблицы, цитаты, код) и HTML-теги
  `<details>/<summary>`.
- Лимит одного rich-сообщения — 32 768 символов; режем с запасом по 30 000
  (`RICH_MESSAGE_MAX_LEN`), деление по границам блоков — пустым строкам
  (`split_markdown_chunks`).
- **Фолбэк**: если `sendRichMessage` вернул ошибку — обычный `sendMessage` с
  `parse_mode: HTML` (конвертация `convert_md_to_html_fallback`: bold/italic/code/ссылки),
  кусками по ~3 800 символов (лимит 4 096).

## 3. Структура поста-выжимки

Пост собирает `build_post_from_document` из тела конспекта:

```
# Название документа            ← крупный H1, остаётся открытым сверху
<текст TL;DR>                   ← суть без ярлыка «TL;DR», открыта сверху

<details><summary><b>Ключевые идеи</b></summary> … </details>
<details><summary><b>Раздел выжимки 1</b></summary> … </details>
<details><summary><b>Раздел выжимки 2</b></summary> … </details>
<details><summary><b>Источники</b></summary> … </details>

#хэштеги                        ← только в архивном канале
```

Правила преобразования (порядок как в коде):

| Шаг | Что делает | Где |
|---|---|---|
| 1 | Срезает YAML front-matter | `build_post_from_document` |
| 2 | Удаляет раздел «Оглавление/Содержание» целиком — его роль играют свёрнутые плашки | там же |
| 3 | Удаляет заголовок `## TL;DR`, сам текст сути остаётся открытым сверху | там же |
| 4 | Вики-ссылки `[[Имя]]` → `**Имя**` (в TG не кликабельны) | там же |
| 5 | Каждый заголовок `##`–`######` → плоская плашка `<details><summary>…</summary>` | `build_accordion_md` |
| 6 | Заголовок плашки — жирным (`<b>`), из него вычищается Markdown-разметка | там же |
| 7 | Заголовки-контейнеры без собственного текста (напр. «Выжимка») опускаются | там же |

Принципиальные ограничения аккордеона:

- **Вложенных `<details>` нет** — вложенность ломает структуру в Telegram,
  аккордеон всегда плоский (каждый заголовок любого уровня = плашка верхнего уровня).
- Заголовки внутри code-fence (```` ``` ````) плашками не становятся.
- **Mermaid запрещён везде** (не рендерится в TG); Markdown-таблицы разрешены
  в выжимках, запрещены в вики.

## 4. Разметка внутри контента

Разметку контента задают правила стиля (`vault/AGENTS.md`, раздел 2) — агент пишет
конспект сразу «под Telegram»:

- короткие абзацы по 2–4 предложения через пустую строку; перечисления — буллетами;
- нумерованный список для «Ключевых идей»;
- Markdown-таблицы для сравнений и характеристик;
- термины с пояснением в скобках, без цитат и таймкодов.

## 5. Служебные сообщения

- **Статус-сообщение** — обычный `sendMessage` (HTML), редактируется по стадиям:
  `⏳ анализ → 📖 читаю → ✍️ конспектирую → 📤 доставка → ✅ готово`.
  После успеха к нему привязывается сессия агента: reply на это сообщение продолжает диалог.
- **Уведомления агента** (`RESULT: NONE`: ошибка загрузки, подтверждение краулинга) —
  HTML-сообщение «ℹ️ Уведомление агента: …».
- Ответы `/ask` — rich-сообщение напрямую из текста агента (без аккордеона),
  с HTML-фолбэком.

## 6. Известные грабли

- `RESULT:`/`SUMMARY:` маркеры парсятся построчно — текстовые блоки агента
  склеиваются через `\n` (`agent.py`), иначе маркер прилипает к предыдущей фразе.
- `<summary>` рендерится мельче основного текста — поэтому заголовки плашек жирные.
- Литеральный HTML в тексте конспекта не экранируется при сборке аккордеона —
  агенту нельзя писать сырые `<теги>` в выжимку.
