# Рич-форматирование ответов LOCUS

Как устроены посты бота в Telegram: структура ответа, применяемая разметка и путь
от raw-конспекта до сообщения. Код: `bridge/telegram_out.py`, `bridge/archive.py`,
`bridge/main.py` (`build_digest_from_file`, доставка в `execute_task`).

## 1. Каналы доставки

Один raw-конспект (`vault/raw/YYYY-MM-DD-slug.md`) рождает два поста:

1. **Чат владельца** — выжимка текстом (`build_digest_from_file` → `send_markdown_text`).
2. **Архивный канал** (если задан `ARCHIVE_CHANNEL_ID`) — та же выжимка + строка
   хэштегов из front-matter (`#type` + `#tags`), собирается в `archive.build_channel_post`.

Файлы `.md` в Telegram **не отправляются**: Bot API отдаёт их с кривым MIME
(расширения нет в таблице TDLib), предпросмотр ломается. Отправка файла осталась
только как фолбэк, если выжимку не удалось собрать.

## 2. Транспорт: rich-сообщения Bot API 10.1

- Единственный путь — `sendRichMessage` с полем `rich_message: {"markdown": "<текст>"}`:
  Telegram сам парсит Markdown (заголовки, списки, таблицы, цитаты, код) и HTML-теги
  `<details>/<summary>`. **HTML-фолбэка нет** (убран по решению владельца, 2026-07):
  ошибка отправки логируется и сообщается, но не деградирует пост в plain-HTML.
- Деление на части «— Ч.N» — только когда пост не влезает в лимит rich-сообщения
  (32 768): порог `POST_PART_MAX_LEN = 31 500`, режем по границам плашек аккордеона,
  не влезающая целиком плашка уходит в следующее сообщение. TL;DR-преамбула — только
  в первой части, шапка с названием капсом — в каждой.
- Страховка при отправке: `split_markdown_chunks` с `RICH_MESSAGE_MAX_LEN = 32 000`,
  деление по границам блоков — пустым строкам (срабатывает только на вырожденных
  случаях вроде одной гигантской плашки).
- **Галерея кадров видео (`+frames`)** добавляет второй режим транспорта поверх
  того же `sendRichMessage`, `send_rich_message` в `bridge/telegram_out.py`:
  - без галереи (99% сообщений — статусы, `/ask`, посты без кадров) — путь как
    описан выше, чистый JSON, без изменений;
  - первая отправка галереи (локальные JPEG из `vault`/`_DATA_DIR/gallery/<slug>/`) —
    `multipart/form-data`: `rich_message` едет строкой (`json.dumps`) в поле `data`,
    сами картинки — отдельными частями `files` под ключами `shot1..shot5`,
    `InputRichMessageMedia.media` ссылается на них через `attach://shot1` и т.д.;
  - повторная отправка той же галереи (архивный канал, см. §1) — чистый JSON,
    `media` ссылается уже на выданные Telegram `file_id`, повторной заливки файлов нет.
  - Ответ Bot API рекурсивно обходится (`extract_gallery_file_ids`) — собираются
    `file_id` всех фото по `file_unique_id` (берём размер с максимальным `file_size`),
    порядок соответствует порядку `shot1..shotN` в галерее.

## 2a. Структура поста при `+frames`: галерея первым блоком

Кадры видео (см. `prompts/ingest_video.md`, разбор `gallery:` во front-matter)
превращаются в блок `<tg-slideshow>` (`build_gallery_md`, `bridge/telegram_out.py`) —
он ставится в САМОЕ НАЧАЛО первой части поста, ДО шапки с названием:

```
<tg-slideshow>

![](tg://photo?id=shot1 "Подпись кадра 1")
![](tg://photo?id=shot2 "Подпись кадра 2")
…

</tg-slideshow>

---

# НАЗВАНИЕ ДОКУМЕНТА
…
```

Telegram рендерит это в порядок блоков: слайд-шоу → дивайдер → заголовок → дивайдер →
цитата TL;DR → аккордеон. Галерея — только в первую часть длинного поста («Ч.2» и
далее её не несут, см. `build_post_parts(gallery_captions=...)`); длина блока галереи
вычитается из бюджета именно первой группы при делении на части, чтобы она не переполнилась.
От 1 до `GALLERY_MAX = 10` кадров на пост — конкретное число решает агент
по визуальной ценности материала (не берёт кадры ради галочки, см.
`prompts/ingest_video.md`).

## 3. Структура поста-выжимки

Пост собирает `build_post_from_document` из тела конспекта:

```
# Название документа            ← крупный H1, остаётся открытым сверху
<текст TL;DR>                   ← суть без ярлыка «TL;DR», открыта сверху

<details><summary><b>Ключевые идеи</b></summary> … </details>
<details><summary><b>Раздел выжимки 1</b></summary> … </details>
<details><summary><b>Раздел выжимки 2</b></summary> … </details>
<details><summary><b>Источники</b></summary> … </details>

#хэштеги                        ← только в архивном канале
```

Правила преобразования (порядок как в коде):

| Шаг | Что делает | Где |
|---|---|---|
| 1 | Срезает YAML front-matter | `build_post_from_document` |
| 2 | Удаляет раздел «Оглавление/Содержание» целиком — его роль играют свёрнутые плашки | там же |
| 3 | Удаляет заголовок `## TL;DR`, сам текст сути остаётся открытым сверху | там же |
| 4 | Вики-ссылки `[[Имя]]` → `**Имя**` (в TG не кликабельны) | там же |
| 5 | Каждый заголовок `##`–`######` → плоская плашка `<details><summary>…</summary>` | `build_accordion_md` |
| 6 | Заголовок плашки — жирным (`<b>`), из него вычищается Markdown-разметка | там же |
| 7 | Заголовки-контейнеры без собственного текста (напр. «Выжимка») опускаются | там же |

Принципиальные ограничения аккордеона:

- **Вложенных `<details>` нет** — вложенность ломает структуру в Telegram,
  аккордеон всегда плоский (каждый заголовок любого уровня = плашка верхнего уровня).
- Заголовки внутри code-fence (```` ``` ````) плашками не становятся.
- **Mermaid запрещён везде** (не рендерится в TG); Markdown-таблицы разрешены
  в выжимках, запрещены в вики.

## 4. Разметка внутри контента

Разметку контента задают правила стиля (`vault/AGENTS.md`, раздел 2) — агент пишет
конспект сразу «под Telegram»:

- короткие абзацы по 2–4 предложения через пустую строку; перечисления — буллетами;
- нумерованный список для «Ключевых идей»;
- Markdown-таблицы для сравнений и характеристик;
- термины с пояснением в скобках, без цитат и таймкодов.

## 5. Служебные сообщения

- **Статус-сообщение** — обычный `sendMessage` (HTML), редактируется по стадиям:
  `⏳ анализ → 📖 читаю → ✍️ конспектирую → 📤 доставка → ✅ готово`.
  После успеха к нему привязывается сессия агента: reply на это сообщение продолжает диалог.
- **Уведомления агента** (`RESULT: NONE`: ошибка загрузки, подтверждение краулинга) —
  HTML-сообщение «ℹ️ Уведомление агента: …».
- Ответы `/ask` — rich-сообщение напрямую из текста агента (без аккордеона),
  с HTML-фолбэком.

## 6. Арсенал форматирования (по исследованию официальных источников, 2026-07)

### Только в rich-сообщениях (Bot API 10.1, 2026-06-11)

Блочные (`RichBlock*`, 21 тип): параграф, заголовки секций, код, футер-сноска,
**разделитель (divider)**, LaTeX-математика, якоря, списки (в т.ч. вложенные),
цитата, pull-цитата, коллаж/слайд-шоу, **таблицы с colspan/rowspan/align**,
**details (аккордеон, поле `is_open`)**, карта, медиаблоки,
thinking-блок (только в стриминг-черновиках `sendRichMessageDraft`).

Инлайновые (`RichText*`, 25 типов): помимо классики — **highlight (marked)**,
sub/superscript, инлайн-формулы, **hashtag как отдельный тип**, сноски-референсы,
якорные ссылки, дата/время, банковские карты.

Важные факты (частично по community-источникам, помечены в отчётах агентов):
- `summary` у details — тип RichText: **только инлайн-разметка** (жирный, курсив,
  эмодзи, ссылки), блочные элементы внутри нельзя.
- Лимиты: ~32 768 символов, ~500 блоков, глубина вложенности ~16, до 20 колонок
  в таблице; длинные посты (>~8 000 симв.) клиент прячет за кнопкой «Show more».
- Заголовки H1–H2 рендерятся жирным с подчёркиванием, H3–H4 жирным, H5–H6 курсивом;
  H1–H4 могут получать авто-эмодзи-префикс (не подтверждено первоисточником).
- Известный баг клиентов: тело rich-сообщения рендерится крупнее обычных сообщений
  и игнорирует пользовательский размер шрифта — вероятно, отсюда «основной шрифт
  больше подзаголовков».
- `sendRichMessageDraft` — эфемерный стриминг (~30 с), только приватные чаты;
  фиксация — финальным `sendRichMessage`.
- `editMessageText` умеет параметр `rich_message`; без него редактирование
  деградирует rich-пост в плоский текст.

### Классический parse_mode (HTML/MarkdownV2) — для фолбэка и статусов

- Entities: bold, italic, underline, strikethrough, **spoiler**, code, pre с языком
  (`<pre><code class="language-x">`), ссылки, custom emoji, blockquote,
  **expandable blockquote** (`<blockquote expandable>` — цитата, свёрнутая до
  3 строк). `code`/`pre` не комбинируются ни с чем; цитаты не вкладываются.
- Лимиты: 4 096 символов текста, 1 024 — caption; ошибка разметки =
  `400 can't parse entities` (сообщение не уходит вовсе).
- Вокруг текста: `LinkPreviewOptions` (превью над текстом / малое / отключить /
  выбрать URL), inline-кнопки (url, callback, **copy_text**), реакции бота,
  `sendChatAction`, message effects (только приватные чаты).

## 7. Идеи улучшения выжимки (бэклог)

Отсортировано по соотношению эффект/риск. Разделитель TL;DR/аккордеон и
эмодзи-иконки заголовков плашек — уже реализованы (см. §3, `_summary_icon`);
свайп-галерея кадров — тоже реализована (§2, §2a). Убраны из бэклога.

1. **Highlight (marked) для ключевой фразы TL;DR** — визуальный якорь поста без
   утяжеления структуры.
2. **`is_open` у первой плашки** — «Ключевые идеи» можно отдавать раскрытыми
   по умолчанию, остальные свёрнутыми: читатель сразу видит скелет материала
   (рассматривалось и отклонено для самой плашки «Ключевые идеи» — она свёрнута
   как и остальные, см. коммит «key-ideas plaque collapsed like the rest»).
3. **Инлайн-акценты в теле разделов** — метрики/названия инструментов через `code`,
   важные выводы через highlight; дёшево поднимает сканируемость.
4. **Таблицы для сравнений активнее** — colspan/rowspan/выравнивание уже
   поддерживаются; правило стиля может прямо предлагать таблицу для «до/после»,
   «плюсы/минусы», версий.
5. **Inline-кнопки под постом** — «Оригинал» (url) вместо раздела «Источники»
   в чате; `copy_text` с чистым текстом TL;DR для быстрой пересылки.
6. **Link preview с `show_above_text`** — превью первоисточника как обложка поста
   (классический sendMessage-путь, применимо к архивному каналу).
7. **Улучшить HTML-фолбэк** — expandable blockquote как замена аккордеона
   (свёрнутые секции доступны и без rich-сообщений), pre с подсветкой языка.
8. **Стриминг черновика выжимки** (`sendRichMessageDraft` + thinking-блок) —
   показывать генерацию в реальном времени вместо статус-сообщения; только для
   личного чата, требует финальной фиксации sendRichMessage.

Отдельно — **гипотеза по багу крупного шрифта**: клиенты рендерят rich-тело крупнее
обычных сообщений независимо от нашей разметки. Если подтвердится, «уменьшить
основной шрифт» изнутри поста нельзя — можно лишь смешивать: прозу слать обычным
sendMessage, а rich использовать только там, где нужны details/таблицы.

## 8. Известные грабли

- `RESULT:`/`SUMMARY:` маркеры парсятся построчно — текстовые блоки агента
  склеиваются через `\n` (`agent.py`), иначе маркер прилипает к предыдущей фразе.
- `<summary>` рендерится мельче основного текста — поэтому заголовки плашек жирные.
- Литеральный HTML в тексте конспекта не экранируется при сборке аккордеона —
  агенту нельзя писать сырые `<теги>` в выжимку.
