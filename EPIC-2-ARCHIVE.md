# EPIC-2: LOCUS Archive — доступ к MD-документам из Telegram

**Версия:** 1.0 · **Дата:** 2026-07-13 · **Статус:** ✅ реализован · **Зависит от:** EPIC.md (основной эпик), Stories 5–6 (бридж + доставка)

> Самодостаточный документ: новая сессия может выполнить этот эпик, прочитав его и разделы 0–3 основного `EPIC.md` (контекст проекта, контракты, env). Основной эпик описывает пайплайн «ссылка → конспект → вики»; этот эпик добавляет поверх него **архив всех результатов, доступный владельцу из Telegram**.

---

## 1. Цель

Владелец должен иметь постоянный доступ из Telegram ко **всем** MD-документам — результатам каждой обработки (конспектам из `vault/raw/`). Архив = хронологическая лента + поиск + точечное получение любого документа файлом. Источник истины — `vault/raw/`; Telegram — витрина.

## 2. Дизайн: два слоя

### Слой 1 — архивный канал (пассивный, автоматический)

Приватный Telegram-канал **«LOCUS Archive»**; бот — администратор с правом постить. После каждого успешного ingest бридж публикует в канал один пост:

```
<TL;DR из SUMMARY>

#article #ai_agents #karpathy        ← тип + теги из front-matter документа
📄 2026-07-13-nazvanie.md            ← прикреплённый .md-файл (sendDocument)
```

Что это даёт без кода: вечное хранение файлов на серверах Telegram, хронологическую ленту, нативный поиск Telegram по тексту и тегам, доступ с любого устройства. Рабочий чат с ботом остаётся чистым диалогом.

Механика: обычный `sendDocument` в `chat_id = ARCHIVE_CHANNEL_ID` (отрицательный id вида `-100…`); caption = SUMMARY + строка тегов (лимит caption 1024 символа — обрезать SUMMARY при необходимости). Rich Message в канал не нужен — файл + caption достаточно.

### Слой 2 — команды по vault (активный, без LLM)

Бридж отвечает на команды **сам, файловыми операциями** — сессия агента НЕ создаётся, лимиты Pro НЕ тратятся:

| Команда | Поведение |
|---|---|
| `/archive [N]` | Последние N (дефолт 10) документов из `vault/raw/` по дате: «заголовок · дата · тип», каждый с инлайн-кнопкой «📄 файл» (callback → sendDocument) |
| `/find <слова>` | Регистронезависимый текстовый поиск по содержимому `vault/raw/*.md` → до 10 совпадений (заголовок + сниппет строки) с кнопками «📄 файл» |
| `/get <slug>` | Документ по имени/части имени файла → sendDocument; при неоднозначности — список кандидатов с кнопками |
| `/backfill` | Разовая догрузка: запостить в архивный канал все документы из `raw/`, которых там ещё нет (для документов, созданных до внедрения этого эпика). Защита от дублей — реестр (см. 3) |

Смысловой поиск «по содержанию базы» — это `/ask` из основного эпика (Story 8, через агента); здесь только дешёвый механический доступ.

## 3. Контракты и данные

- **Точка интеграции**: в бридже после успешного парсинга `RESULT: <path>` (контракт 3.1 основного эпика) — вызов `archive_post(path, summary)`. Ошибка постинга в канал НЕ должна ронять основную доставку владельцу (лог + продолжить).
- **Теги**: тип документа (`#video`/`#article`/`#text`) из front-matter `type`; тематические — из front-matter `tags` (транслитерация/подчёркивания: `ai agents` → `#ai_agents`).
- **Реестр отправленного**: `bridge/archive_index.json` — `{ "<имя файла>": {"message_id": ..., "posted_at": ...} }`. Нужен для `/backfill` (не дублировать) и на будущее (редактирование/удаление постов). Файл в `.gitignore` не нужен — это не секрет, но и не обязателен в git.
- **env**: `ARCHIVE_CHANNEL_ID` в `bridge/.env` (заготовка уже добавлена). Получение id: переслать любой пост канала боту @userinfobot либо временно залогировать `getUpdates` после поста в канал.
- Команды принимаются только от `OWNER_CHAT_ID` (та же проверка, что во всём бридже).

## 4. Стори

### Story A1 — Автопостинг в архивный канал
- [x] Владелец создаёт приватный канал, добавляет бота админом, id → `ARCHIVE_CHANNEL_ID`.
- [x] `bridge/archive.py`: `archive_post(doc_path, summary)` — чтение front-matter (type, tags), сборка caption (SUMMARY + теги, обрезка до 1024), `sendDocument` в канал, запись в `archive_index.json`.
- [x] Вызов из пайплайна доставки после успешного `RESULT:`; ошибки не блокируют ответ владельцу.

Приёмка: обработка тестовой ссылки → пост в канале с файлом, TL;DR и тегами; поиск Telegram по тегу находит пост.

### Story A2 — Команды доступа
- [x] `/archive [N]` — листинг из front-matter файлов `raw/` (title, date, type), инлайн-кнопки, callback-обработчик → sendDocument.
- [x] `/find <слова>` — поиск по содержимому, сниппеты, кнопки.
- [x] `/get <slug>` — точечная выдача, обработка неоднозначности.
- [x] Всё без вызова Agent SDK; отклик < 2 сек на vault в сотни документов.

Приёмка: каждая команда возвращает корректные результаты на vault с ≥3 документами; чужой chat_id — отказ.

### Story A3 — Backfill и устойчивость
- [x] `/backfill` — обход `raw/`, постинг отсутствующих в `archive_index.json`, троттлинг ~1 сообщение/3 сек (лимиты Telegram на постинг в канал ~20 сообщений/мин).
- [x] Ретрай постинга ×1 при сетевой ошибке; при 429 от Telegram — уважать `retry_after`.
- [x] Обновить README и раздел 0 основного EPIC.md: архив реализован, где лежит реестр.

Приёмка: на vault с существующими документами `/backfill` догружает все и не дублирует при повторном запуске.

## 6. Риски

| Риск | Митигция |
|---|---|
| Лимиты Telegram на постинг в канал (при backfill) | Троттлинг + retry_after (A3) |
| Caption > 1024 символов | Обрезка SUMMARY с многоточием |
| Файл > 50 МБ (лимит sendDocument для ботов) | Нереально для .md-конспектов; на всякий случай — проверка размера и сообщение об ошибке |
| Потеря archive_index.json | Не критично: /backfill может сверяться и по истории канала в будущем; дубль в канале — косметическая проблема |
